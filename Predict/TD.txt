League Insights — Trajectory prediction (research note)
=======================================================

Introduction
------------
We study short-horizon prediction of player position in League of Legends from
match timelines. Full-map coordinates are noisy and hard to regress directly;
we therefore formulate the task as discrete sequence classification: given recent
game state, estimate a probability mass over map quadrants (or grid cells) for
where the player will be at the next timestep (or over a small lookahead). A
single “best” cell is optional—typically argmax or top-k from that distribution—
because in reality several regions are often plausible. Data are assumed to come
from a merged timeline of regular frame snapshots plus sparse in-game events, so
each training example is a row or sequence ending at time t and a label derived
from the next chronological state in that merged stream.

What we are measuring
---------------------
Primary: how well the full predictive distribution matches held-out labels—
cross-entropy / log loss (and related proper scoring rules), which reward
assigning mass to the true cell and to plausible neighbors if we add spatial
smoothing or distance-weighted targets. Secondary: argmax accuracy, per-class
recall, and macro-F1 (these only reflect the peak, not spread mass). We also
check calibration of probabilities (reliability curves or post-hoc calibration
if decisions depend on numeric confidence), distance-aware summaries (e.g.
expected error under a map metric), and comparison against naive baselines
(persistence, Markov / n-gram on quadrant sequences). Confusion matrices on
argmax still help show whether errors cluster on spatial neighbors.

What we intend to do
--------------------
Learn a model that maps recent trajectory and scalar game features to a
normalized distribution over discrete locations (softmax over cells). Treat that
vector as the main output for analysis and downstream use (e.g. top-k regions,
entropy as uncertainty); derive a point prediction only when the product needs
one. Start with the simplest targets and baselines and scale complexity only if
needed. Champion identity and coarse playstyle categories (see champCategory.py
for jungler-oriented groupings) may enter as categoricals or embeddings; role,
team side, and game phase are candidates for additional conditioning.

Formal task definition (enemy jungler pathing first)
----------------------------------------------------
Primary scope for v1 is enemy-jungler path prediction during laning phase.
We define:

- r_t: the current row in the merged per-jungler sequence.
- t: current row timestamp in milliseconds.
- h_t: the row-specific prediction horizon, stored as `targetHorizonMs`.
- T_lane: laning cutoff (default T_lane = 14 * 60 * 1000 ms).
- x_t: feature vector at row r_t.
- y_t: supervision target at the next chronological row r_(t+1), represented as
  continuous coordinates/displacements or a discretized map cell.
- m_t in {0,1}: validity mask, where m_t = 1 iff `targetX` and `targetY` are not
  null and `timestampMs <= T_lane`.

Per-row inclusion criterion:
  include row r_t <=> m_t = 1.

Target options:
1) Continuous regression:
   y_t = [targetX, targetY] or [targetDx, targetDy].
2) Discrete classification:
   y_t = c_t where c_t is a map cell / quadrant index derived from targetX/Y.

Important dataset detail: the raw extractor no longer forces a fixed 60-second
label. Instead, every row points to the next chronological row in the merged
sequence. For plain `frame -> frame` transitions this is usually about one
minute, but event rows (`kill`, `death`, `backing`, etc.) create shorter,
semantically richer hops.

The initial project decision is to start with enemy-jungler pathing only,
then expand to recommendation tasks after the enemy-location baseline is stable.

Feature partition and notation
------------------------------
We partition features into three groups:

- Data evidence features D_t:
  position, level/cs/gold state, movement speed, row type, side/champion, and
  sparse event semantics from the merged sequence.
- Style-prior features P_t:
  champion playstyle category and suggested-clear-path signals
  (e.g., distance to suggested camp, clear prior score).
- Context features C:
  static match/champion metadata (when available and leakage-safe for the
  specific task phase).

Model input:
  x_t = concat(D_t, P_t, C).

Important leakage rule:
- End-of-game outcomes (e.g., team win) are never fed as inference-time inputs.
- They may be used only for offline sample weighting (see below).

Laning-phase gating and masks
-----------------------------
Training mask:
  m_t = 1[targetX_t != null and targetY_t != null and timestampMs_t <= T_lane]

Effective sample set:
  S = {t | m_t = 1}

All losses and metrics are computed over S unless explicitly stated otherwise.

Weighted objective (data quality and product alignment)
-------------------------------------------------------
We use a multiplicative per-sample weight:

  w_t = w_win(t) * w_rank(t) * w_conf(t)

where:
- w_win(t): optional win/loss reliability weight for the jungler team
  (for imitation-style bias toward successful trajectories).
- w_rank(t): optional rank-bucket emphasis weight
  (for deployment-target alignment without hard rank splitting).
- w_conf(t): optional confidence/quality factor
  (e.g., stronger sightings, lower missingness, better event support).

Recommended starting point:
- w_win in {1.0, 1.25, 1.5} for won rows and 1.0 for lost rows.
- w_rank near 1.0 for all buckets unless a target audience is explicit.
- keep w_conf = 1.0 initially.

Loss for regression (Huber):
  L_reg = (sum_{t in S} w_t * huber(y_t, yhat_t)) / (sum_{t in S} w_t)

Loss for classification (cross-entropy):
  L_cls = (sum_{t in S} w_t * CE(c_t, phat_t)) / (sum_{t in S} w_t)

where phat_t is predicted probability over cells.

Prior-vs-data blending (clear-path priors)
------------------------------------------
To encode early-game uncertainty, we treat recommended clear routes as a soft
prior and match-derived features as evidence.

Let:
- s_prior_t in [0,1]: prior score (e.g., from expected-camp distance).
- alpha_prior >= 0: prior weight.
- alpha_data >= 0: data-evidence weight.

A simple blended feature:
  b_t = alpha_prior * s_prior_t + alpha_data * 1.0

This b_t is provided to the model as an input (not as a hard constraint), so
the network can override priors when evidence contradicts them.

Alternative probabilistic view (for later versions):
  p(y_t | x_t) propto p_data(y_t | D_t, C)^(alpha_data) * p_prior(y_t | P_t)^(alpha_prior)

This formulation makes the role of prior weighting explicit and tunable.

Side canonicalization and symmetry
----------------------------------
Default training policy is one shared model over both blue and red side after
coordinate canonicalization (mirror transform to one reference perspective).
The same transform must be applied consistently to:
- current position,
- targets,
- event coordinates used as inputs.

This reduces duplicated side-specific geometry learning and improves sample
efficiency. Side flags can be retained as residual features and ablated.

Rank strategy and pooling
-------------------------
Default strategy is pooled training across ranks to maximize sample size and
avoid over-fragmentation. Rank is treated as:
- optional conditioning feature, and/or
- mild weighting factor for deployment alignment.

Hard per-rank model splitting is not default due to reduced effective data and
higher variance.

Evaluation protocol (publication-grade)
---------------------------------------
Split discipline:
- hold out by matchId (or by player) to prevent temporal/identity leakage.
- never split individual rows from the same match across train/val/test.

Core metrics:
- regression: mean/median distance error, radius-hit rates.
- classification: log-loss, top-1/top-k accuracy, calibration.

Stratified reports (mandatory):
- by rank bucket,
- by patch (`gameVersion`),
- by side handling (raw vs canonicalized),
- optionally by champion or playstyle category.

Ablation matrix (minimum):
- remove prior block P_t.
- remove motion block (dx/dy/speed).
- remove event block.
- remove gold-derived features.
- change alpha_prior / alpha_data.
- toggle win/rank weights.

A change is accepted only when aggregate and key stratified metrics improve or
trade-offs are explicitly justified.

How we will achieve it (implementation plan)
--------------------------------------------
1. Data construction  
   Build supervised samples: features at time t (and optionally a short lag
   window) from time, position history, CS, level, champion ids, and
   role-aligned opponents; label = quadrant at t + h. Avoid leakage: no
   future CS, kills, or events not observable at t. Optionally normalize map
   coordinates to one team perspective or augment with mirror flips to reduce
   blue/red artifacts.    Collection policy: fetch up to 100 matches per player
   (or fewer if unavailable), restrict to Ranked Solo/Duo on Summoner's Rift
   (queueId 420, mapId 11), and deduplicate by matchId across players so one
   unique match contributes all 10 participant timelines without duplicate files.
   Planning numbers: `fetchMatch.py` defaults to **`--count 100`** (max match IDs
   requested from each player's match history per run). In practice, bulk
   download throughput is on the order of **~20 completed matches per minute per
   development API key** (each unique match is match JSON plus timeline JSON, so
   rate limits dominate). The corpus we are aiming for is on the order of
   **~700,000 unique matches** after cross-player deduplication—not 100× players
   raw, because many accounts share the same games.

   Repository `Data/` flow (bulk): `fetchPlayer.py` writes ladder JSONL under
   `Data/players/<run>/` with **`username`** and **`tag`** per line; `fetchMatch.py`
   merges players by Riot ID, resolves **puuid** via the Account API, downloads
   flat `Data/matches/<run>/` and `Data/timelines/<run>/`, and **skips** a matchId
   when raw files already exist or when processing already produced either
   `Data/log/<run>/.done/<matchId>` or grouped log files under
   `Data/log/<run>/<ChampionName>/<matchId>_p<participantId>_<side>.json`.
   `processMatch.py` reads each match/timeline pair and writes one pretty JSON
   per jungler under the champion folder. After successful processing it touches
   the empty `.done/<matchId>` marker and removes the raw match/timeline pair.

   The current on-disk dataset contract is therefore:
   `Data/log/<run>/<ChampionName>/<matchId>_p<participantId>_<blue|red>.json`
   where each file contains one jungler only.

Extracted features (current schema)
-----------------------------------
Source: `Data/processMatch.py` -> each processed file is
`Data/log/<run>/<ChampionName>/<matchId>_p<participantId>_<side>.json`.

A) File header (constant per file)

  Field                      Contents
  -------------------------  --------------------------------------------------
  matchId                    Riot match id string.
  participantId              Jungler participant id for this file.
  teamId                     100 or 200.
  isBlueSide                 Boolean convenience flag (`teamId == 100`).
  championId, championName   This jungler's champion.
  enemyChampionId,
  enemyChampionName          Opposing jungler champion identity.

B) rows[] common fields (present on frame rows and event rows)

  Field                      Contents
  -------------------------  --------------------------------------------------
  rowType                    One of `frame`, `kill`, `assist`, `death`,
                             `respawn`, `dragon`, `plate`, `backing`
  frameIndex                 Unique sequential index in the merged row stream
  timestampMs                Row timestamp in milliseconds
  x, y                       Position for this row (event position or base)
  level                      Level at/just before this row
  targetHorizonMs            Milliseconds to the next chronological row
  targetX, targetY           Position of the next chronological row
  targetDx, targetDy         Displacement from current row to next row

  Terminal rows have null `target*` fields because there is no next row.

C) `frame` row fields

  Field                      Contents
  -------------------------  --------------------------------------------------
  csJungle                   Jungle CS at that frame
  csLane                     Lane CS at that frame
  currentGold                Unspent gold at that frame
  totalGold                  Total earned gold at that frame
  movementSpeed              Current movement speed at that frame

  The extractor intentionally keeps the frame schema small. Repeated metadata is
  lifted into the file header rather than copied onto every row.

D) Event row semantics

  Row type                   Rule / source
  -------------------------  --------------------------------------------------
  kill                       `CHAMPION_KILL` where this jungler is killer
  death                      `CHAMPION_KILL` where this jungler is victim
  assist                     `CHAMPION_KILL` where this jungler assisted
  dragon                     `ELITE_MONSTER_KILL` on a dragon with jungler
                             involved as killer or assist; includes
                             `monsterSubType`
  plate                      `TURRET_PLATE_DESTROYED` where this jungler is
                             killer; includes `laneType`
  backing                    `ITEM_PURCHASED` for this jungler after leaving
                             base; purchases within 15 seconds collapse into one
                             shop visit
  respawn                    Inferred base return after each `death`, using
                             level-based respawn timing

E) Ordering and supervision

  Rows are sorted chronologically, with a stable type order for ties. After the
  merge, `frameIndex` and all `target*` fields are recomputed so every row
  points to the next chronological row in the final sequence. This is the key
  difference from the older flat logs, where supervision was framed only as
  "current frame -> next fixed 60 second frame."

Legacy note:
  Older flat `Data/log/<run>/<matchId>.jsonl` files can be converted only as a
  best-effort backfill. They usually contain enough information for `frame`,
  `kill`, `assist`, `death`, `dragon`, and inferred `respawn`, but they do not
  preserve purchase events needed for the current `backing` rows and may also
  miss plate events.

Modeling notes: weights, motion, variable horizons, and sparse events
---------------------------------------------------------------------
These notes turn product intuition into experiment knobs (losses, weights,
extra features). Nothing here requires changing the extractor unless we add
new derived columns later.

1) Sample weights: wins vs losses
   Idea: rows from matches where the jungler's team won get higher weight than
   losses, so the model prefers trajectories that correlate with winning play.
   This should be an offline weighting only, never a live input feature.

2) Motion is now implicit, not precomputed
   The current extractor no longer writes `dx1`, `dy1`, or speed deltas. If a
   model wants short-history motion features, derive them in the dataloader from
   consecutive `frame` rows or from the full merged sequence. This keeps the
   on-disk schema lean and makes it easy to change lag windows later.

3) Level timing and jungle CS
   `level` and `csJungle` remain strong signals for path phase and clear tempo.
   Even without explicit XP columns, their progression over successive rows helps
   distinguish full-clear, invade, reset, and fight-heavy branches.

4) Gold and explicit back timing
   `currentGold` and `totalGold` remain useful state features, but `backing` is
   no longer inferred from gold drops. The extractor now uses `ITEM_PURCHASED`
   events directly, which is strictly better for recall/shop timing.

5) Sparse event rows are first-class signals
   Instead of collapsing nearby events into coarse counts/distances, the current
   dataset stores sparse rows directly in the main sequence. This makes event
   transitions explicit and lets models learn patterns such as
   `frame -> kill -> backing -> frame` or `frame -> death -> respawn -> frame`.

6) Variable horizon is part of the problem now
   `targetHorizonMs` is no longer a constant label by construction. Downstream
   training can:
   - use it as an input,
   - filter to a narrower horizon band,
   - or resample to custom fixed-horizon tasks.
   The raw log preserves the richer event-aware chronology first.

Quantification (how to know these rules help)
---------------------------------------------
  - Win vs loss weights: grid-search on a validation split by `matchId`; track
    mean distance error or grid log-loss and calibration.
  - Motion features: compare models with only raw positions versus derived
    multi-lag motion features from the dataloader.
  - Level / CS / gold: ablation removing progression/economy blocks versus full
    model.
  - Sparse events: compare `frame`-only models against models that ingest the
    merged event rows or event-derived summaries.
  - Variable horizon handling: compare raw `targetHorizonMs` conditioning versus
    filtered fixed-band training.
  - Rank / side (see below): report metrics stratified by ladder bucket and by
    raw side versus canonicalized side whenever those dimensions exist in the
    dataloader metadata.

Training-data design choices (for publication / reproducibility)
-----------------------------------------------------------------
These are explicit decisions from project discussion; they affect comparability
across experiments and any downstream paper.

A) Blue side vs red side — pool or split?  
   Summoner’s Rift is approximately mirror-symmetric. Raw `(x, y)` without
   handling side forces the model to learn the **same semantic path** in two
   different corners of the map (blue vs red), wasting capacity.

   **Recommended default:** train **one** model on **both** sides, after
   **canonicalizing coordinates** to a single perspective (e.g. map all red-side
   samples through a fixed mirror so “bot side jungle” always lands in the same
   half of the feature space). Apply the **same** transform to:
   - current `(x, y)` and supervision targets (`targetX`, `targetY`, `targetDx`,
     `targetDy`);
   - any event-row coordinates used as inputs or auxiliary tasks.

   The per-row flag `isBlueSide` (or `teamId`) can remain as an **optional**
   residual input **after** mirroring if ablations show real asymmetry (dragon
   pit layout, meta, camera). If mirroring fully explains side effects, drop the
   flag to reduce parameters.

   **Not recommended** unless data-rich and hypothesis-driven: entirely separate
   models per side (splits the dataset and hurts generalization).

B) Rank / ladder — group logs by division or train together?
   On disk, logs now live under
   `Data/log/<run>/<ChampionName>/<matchId>_p<participantId>_<side>.json`; ladder
   tier can be inferred from the run folder name or other metadata if needed.
   **Training does not require** separate datasets per rank bucket.

   **Default:** concatenate **all ranks** when fitting a champion-centric or
   global model, to maximize samples per `championId` and learn coarse pathing
   that is largely rank-invariant (camps, timers, geometry).

   **Caveat — distribution shift:** high-elo games differ in tempo and risk;
   low-elo has more noise. Pooling can yield a model that is **average** across
   skill bands.

   **Mitigations (pick one or combine):**
   1. **Conditioning:** add a categorical ladder bucket (from folder name or
      metadata) or a coarse `skillBin` as an input / embedding. The model can
      learn rank-specific residuals without discarding data.
   2. **Mild sample weights:** up-weight the tier closest to the **deployment**
      audience (e.g. Diamond+), keep weights close to 1.0 elsewhere; tune on
      validation. Avoid extreme weights that make rare tiers dominate gradients.
   3. **Stratified evaluation:** always report held-out error / log-loss **per**
      ladder bucket and per patch (`gameVersion`), not only aggregate metrics.

   **Higher elo = “better labels”?** Not automatically. Top players are more
   consistent in some respects but also play rarer lines; overweighting them
   shrinks effective sample size and can hurt users at other ranks. Treat elo
   weighting as a **hypothesis** to validate, not a default.

C) Champion grouping — one model vs many  
   **Per-champion** models need enough rows; pooling ranks helps. A **single**
   shared model with a `championId` embedding (and summoner spell / patch
   embeddings as needed) usually generalizes better than dozens of tiny
   per-champion nets unless data per champion is very large.

   Record in any write-up: which grouping was used, and whether rank/side
   canonicalization was applied **before** train/val/test splits (it should be
   applied consistently per row, not leaking test matches into normalization).

2. Baselines  
   Persistence (same quadrant as last step) and low-order Markov models on
   quadrant sequences to establish a floor and to judge whether learned models
   add value.

3. Tabular models  
   Gradient-boosted trees (e.g., LightGBM / XGBoost) on engineered lag
   features: recent quadrants, deltas in CS and level, clock or phase. Strong
   default when the horizon is short and history is featurized explicitly.

4. Sequence models (optional)  
   If longer context matters (objective setup, recalls), stack embeddings for
   champions and feed the last T timesteps through a small GRU/LSTM or
   Transformer encoder, then a softmax over quadrants.

5. Training objective and outputs  
   Train with cross-entropy against the true cell (standard multiclass). If we
   want the model to hedge across neighboring cells, optional label smoothing or
   a target distribution that spreads mass to adjacent quadrants can reflect
   spatial ambiguity. Expose predict_proba (or logits → softmax) as the primary
   API; report argmax accuracy as a secondary, user-facing summary.

6. Inputs vs labels  
   Labels discretized to quadrants; inputs may combine raw or fine-grid coordinates with quadrant history so the model retains sub-cell motion where
   useful. Sensitivity to interpolation method should be checked so the model
   does not fit synthetic motion.

7. Evaluation discipline  
   Hold out by match or player where possible; note patch and rank-tier shift
   if training and test distributions differ. Compare against baselines before
   claiming gains from architecture alone.
